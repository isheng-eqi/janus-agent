# Janus CLI 美学设计方案

> 基于 [CLI 界面美学规范研究](./cli-design-research.md) 和 [当前输出调查](./janus-current-output-survey.md)。
> 设计原则：**追求简洁。每减少一行输出都给出理由。**

---

## 一、颜色方案

### 1.1 五色语义调色板

当前 Janus 仅使用 emoji 做语义标记（✅ ❌ ⚠️ 🔍），没有 ANSI 颜色。本方案引入 5 种 ANSI 基础色 + 1 种辅助样式，全部使用 8 种基本 ANSI 颜色（兼容浅色/深色终端主题，NO_COLOR 环境变量自动禁用）。

| 序号 | 颜色 | ANSI 码 | 语义 | 映射场景 |
|------|------|---------|------|----------|
| 1 | **绿色** | `\033[32m` | 成功 / 完成 | 任务通过、审核通过、操作成功 |
| 2 | **红色** | `\033[31m` | 错误 / 失败 | 任务失败、配置错误、运行时异常 |
| 3 | **黄色** | `\033[33m` | 警告 / 注意 | 重试提示、降级行为、Ctrl+C 中断 |
| 4 | **青色** | `\033[36m` | 信息 / 进行中 | 阶段标题、工具调用、任务框标题 |
| 5 | **白色** | 默认 | 正文 | 主体输出、Agent 回复 |

**辅助样式（不引入新颜色）：**

| 样式 | ANSI 码 | 用途 |
|------|---------|------|
| 加粗 | `\033[1m` | 标题、关键数值（如 "3/3 通过"） |
| 暗化/灰色 | `\033[2m` | 次要信息、时间戳、元数据（如 "耗时 3.1s"） |

### 1.2 颜色禁用规则（必须实现）

```
1. stdout 非 TTY（管道到文件或其他程序）
2. NO_COLOR 环境变量已设置
3. TERM=dumb
4. --no-color 命令行标志（新增）
```

### 1.3 每条颜色规则的理由

| 颜色 | 为什么选它 |
|------|-----------|
| 绿色=成功 | 跨文化最通用的"正确/通过"信号色，Docker/Cargo/Git 均如此使用 |
| 红色=错误 | 最强烈的警示色，用户视线会自动被红色吸引——错误信息需要这个效果 |
| 黄色=警告 | 介于"没问题"和"出事了"之间，用于需要用户注意但非致命的场景 |
| 青色=信息 | 与白色正文区分但不制造紧张感；蓝色系在多数终端方案下对比度好 |
| 不加新颜色 | 超过 5 色时颜色失去语义区分力，变成装饰性噪音 |

## 二、输出点位逐项重构

以下按用户看到的时间顺序，对每个输出点位给出「现状 → 问题 → 新设计 → 理由」。

---

### 2.1 启动欢迎信息

**现状（17 行）：**
```
╔══════════════════════════════════════════════════════════════╗
║           Janus — 多轮对话 Agent 框架                        ║
╠══════════════════════════════════════════════════════════════╣
║  只需用自然语言描述你的需求，Janus 会自动拆解并执行。         ║
║                                                            ║
║  示例:                                                      ║
║    > 帮我写一个排序 CSV 文件的 Python 脚本                    ║
║    > 在 ./my-app 下创建 README.md                            ║
║    > 找出代码中所有的 TODO 注释并列出                         ║
║    > 嗨，你能帮我做什么？                                      ║
║                                                            ║
║  命令:                                                      ║
║    help / h / ?  — 显示此帮助信息                            ║
║    quit / exit / q  — 退出 Janus                             ║
╚══════════════════════════════════════════════════════════════╝
```

**新设计（2 行 + 空行）：**
```
Janus · AI Agent 框架
输入「help」了解用法，「quit」退出。

>
```

**每行删除理由：**

| 删除内容 | 行数 | 理由 |
|----------|------|------|
| ASCII 双线框（╔╗╚╝） | 6 行 | 纯装饰，无信息量。用空行 + 排版即可实现视觉分离 |
| 重复的功能说明 | 3 行 | "只需用自然语言描述"在 help 中已覆盖；老用户每次启动都看到它是噪音 |
| 4 条示例 | 8 行 | 示例是帮助内容，不应出现在每次启动中。新用户用 `help` 即可看到 |
| 命令列表 | 4 行 | `help` 和 `quit` 是 CLI 通用惯例，一行提及足矣。`?` 和 `exit`/`q` 等别名无需在启动时列出 |

**保留率：2/17 = 11.8%**。信息密度提高，老用户不再被噪音淹没，新用户仍有一个明确的下一步（输入 `help`）。

---

### 2.2 提示符

**现状（1 行）：**
```
>
```

**新设计：**
```
janus >
```

**理由：** 裸 `>` 无法区分这是 Janus 的 REPL 还是其他工具。加上 `janus ` 前缀提供身份识别——用户切换多个终端窗口时一眼就知道这是 Janus。只增加 5 个字符，不增加行数。当 stdin 非 TTY 时自动降级为无提示符。

---

### 2.3 空输入提示

**现状（1 行）：**
```
  (输入 'help' 查看帮助，'quit' 退出，或按 Ctrl+C 强制中断)
```

**新设计：**
```
  输入 help 或 quit。（Ctrl+C 中断当前任务）
```

**减少 18 个字符的理由：** 原文用 `'help'` 引号 + "查看帮助" + "退出 Janus" + "强制中断" 过度解释，用户只需要知道动作。同时将 Ctrl+C 的语义从"退出"纠正为"中断当前任务"（它中断的是正在执行的任务而非退出 REPL）。

---

### 2.4 REPL 内 `help` 命令

**现状（~18 行）：**
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  Janus 帮助
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Janus 是一个多轮对话 AI Agent 框架。它会将你的目标
  拆解为可执行的任务并逐一完成。

  快速上手：
    • 用自然语言描述任何目标 — Janus 会处理剩下的事。
    • 想闲聊？直接说「你好」或问个问题即可。
    • 输入 'quit' 或按 Ctrl+C 退出。

  技巧：
    • 描述越具体，执行越精准 —「写个脚本」可以，
      「在 ./scripts/ 下写一个读取 data.csv 并输出统计
      结果的 Python 脚本」效果更好。
    • Janus 会读写真实文件 — 执行完后检查你的项目目录。
    • 使用 --verbose 查看详细执行日志。
```

**新设计（~10 行）：**
```
Janus — AI Agent 框架，将自然语言目标拆解为任务并逐一执行。

用法示例：
  > 帮我写一个排序 CSV 的 Python 脚本
  > 在 ./my-app 下创建 README.md
  > 找出所有 TODO 注释

描述越具体，执行越精准。输入 quit 退出，--verbose 查看细节。
```

**每行删除理由：**

| 删除内容 | 行数 | 理由 |
|----------|------|------|
| 双分隔线 | 2 行 | 装饰性分隔，标题自身已足够醒目 |
| 标题单独一行 | 1 行 | 合并到首句——"Janus 帮助"不增加信息 |
| "快速上手"区块标题 | 1 行 | 下面只有 3 条基础信息，无需分区标题 |
| 闲聊提示 | 1 行 | 用户自然会尝试，“想闲聊？”暗示这不是主要功能 |
| "技巧"区块标题 | 1 行 | 内容只有 3 条，分区标题增加层级但无实际帮助 |
| 冗长的技巧示例 | 3 行 | 一条原则性提示比 4 行示例更高效 |

**信息重组逻辑：** help 应该回答三个问题——这是什么？怎么用？有什么技巧？按这个顺序组织，用示例替代说明。

---

### 2.5 Gatekeeper 分解阶段

**现状（2 行 + N）：**
```
🔍 Gatekeeper 分析完成，拆分为 3 个子任务：
  ✓ task-1 · 实现阶乘函数
  ✓ task-2 · 编写单元测试
  ✓ task-3 · 验证测试通过
```

**新设计——标题行改为青色，任务列表保留：**
```
分析完成 · 3 个子任务
  ✓ 实现阶乘函数
  ✓ 编写单元测试
  ✓ 验证测试通过
```

**删除/修改理由：**

| 变更 | 理由 |
|------|------|
| 删除 "Gatekeeper" | 用户不需要知道内部架构角色名——"分析完成"对用户足够 |
| 删除 task-1/task-2/task-3 编号 | 编号在列表对齐中无额外信息，用户更关心任务描述；编号仍在内部保留用于审核引用 |
| 标题简化为 "分析完成 · 3 个子任务" | 原句 20 字 → 9 字，信息量相同 |
| color: cyan | 阶段标题是信息性输出，青色传达"正在推进" |

### 2.6 任务框（task_start）

**现状：**
```
┌─ task-1 · 实现阶乘函数 ───────────────────────────────────────┐
```

**新设计：**
```
┌─ 实现阶乘函数 ──────────────────────────────────────────────────┐
```

**删除 "task-1 ·" 前缀的理由：** 任务 ID 是内部标识，用户看任务描述即可。框的视觉结构本身已区分不同任务。宽度调整时不再需要为 ID 留空间。

**颜色：** 框体使用青色，与"进行分析"的阶段连接起来。

---

### 2.7 工具调用摘要

**现状：**
```
│  ⚡ 写入文件: C:\project\factorial.py
│  ⚡ 执行命令: python factorial.py
```

**新设计——路径用暗化样式：**
```
│  ⚡ 写入文件: factorial.py
│  ⚡ 执行: python factorial.py
│  ⚡ 读取: C:\project\test_factorial.py
```

**修改理由：**

| 变更 | 理由 |
|------|------|
| "执行命令" → "执行" | 用户知道这是在执行命令，缩短 2 字 |
| 路径保持原始格式 | 不做额外处理——长路径自然换行，保持可 grep |
| 颜色：暗化路径 | 文件名是次要信息，暗化让用户视线聚焦在动作上 |

### 2.8 Reviewer 审核通过

**现状：**
```
│
│  🔍 Reviewer 审核中...
│  ✅ 通过
│     ✓ 函数正确地计算了 5! = 120
│     ✓ 代码包含类型提示和文档字符串
│     ✓ 边界条件 n=0 返回 1
```

**新设计：**
```
│  ✅ 通过
│     ✓ 函数正确地计算了 5! = 120
│     ✓ 代码包含类型提示和文档字符串
│     ✓ 边界条件 n=0 返回 1
```

**删除理由：**

| 删除内容 | 行数 | 理由 |
|----------|------|------|
| 空行 | 1 行 | 审核结果紧接工具调用，不需要额外空行 |
| "🔍 Reviewer 审核中..." | 1 行 | 审核通常是瞬时完成（<1s），用户不会看到这行。如果审核确实耗时，由任务框的 spinner 承担进度指示 |

**颜色：** "✅ 通过" 使用绿色，"✓" 证据行保持白色。

---

### 2.9 Reviewer 审核失败 + 重试

**现状：**
```
│
│  🔍 Reviewer 审核中...
│  ❌ 未通过
│     ✗ 函数未处理负数输入
│     ✗ 缺少文档字符串
│  🔄 第 1 次重试...
```

**新设计：**
```
│  ❌ 未通过 — 第 1 次重试
│     ✗ 函数未处理负数输入
│     ✗ 缺少文档字符串
```

**删除理由：**

| 删除内容 | 行数 | 理由 |
|----------|------|------|
| 空行 + "审核中" | 2 行 | 同 2.8 |
| 独立重试行 | 1 行 | 合并到标题行——"未通过 — 第 1 次重试"，用户扫一眼就知道状态和进度 |

**颜色：** "❌ 未通过" 使用红色，"✗" 问题行使用黄色（问题是警告性质的）。

### 2.10 任务完成（task_done）

**现状：**
```
│  ⏱ 耗时 3.1s
└──────────────────────────────────────────────────────┘
  ✅ task-1 · 通过
```

**新设计：**
```
│  ⏱ 3.1s
└──────────────────────────────────────────────────────┘
  ✅ 通过
```

**删除理由：**

| 删除 | 理由 |
|------|------|
| "耗时" | 冗余，⏱ 已传达"时间"语义 |
| "task-1 ·" | 框的视觉分组已标识任务范围，用户不需要编号 |
| "3 个任务" 行 | 任务框边框已足够 |

**颜色：** "✅ 通过" 绿色，"✅ 失败" 红色，⏱ 暗化。耗时数字加粗。

### 2.11 Console 摘要（summary）

**现状：**
```
━━━━━━━━━━━━━━━━━━━━ 汇总 ━━━━━━━━━━━━━━━━━━━━
  ✅ 全部通过: 3/3
```

**新设计（合并到 Gatekeeper 最终汇报，见 2.12）：**

直接删除 `console.summary()`。理由见 2.12。

### 2.12 Gatekeeper 最终汇报 — 全部成功

**现状（~6 行）：**
```
✅ Janus 汇报：3个任务全部完成
  Completed: 3/3 tasks.
  ✅ worker-0: 函数正确实现了阶乘计算
  ✅ worker-1: 测试已编写并通过
  ✅ worker-2: 项目目录结构完整
```

注意到当前有两个汇总——`console.summary()`（2.11）和 `gatekeeper._report_to_user()`（这里），信息重复。合并后：

**新设计（~4 行）：**
```
✅ 完成 · 3/3 个任务
  ✓ 函数正确实现了阶乘计算
  ✓ 测试已编写并通过
  ✓ 项目目录结构完整
```

**删除理由：**

| 删除内容 | 理由 |
|----------|------|
| console.summary() 分隔线汇总 | 与 gatekeeper 汇报重复计数——用户看到两次 "3/3" 是信息重复 |
| "Janus 汇报：" 前缀 | 用户知道这在 Janus 里运行，不需要署名 |
| "Completed: 3/3 tasks." 英文行 | 与中文行完全重复，可能来自 LLM 惯用双语输出 |
| worker-N 编号 | 用户不关心 Worker 实例名 |

**颜色：** "✅ 完成 · 3/3" 使用绿色，3/3 加粗。

---

### 2.13 Gatekeeper 最终汇报 — 全部失败

**现状（~13 行）：**
```
❌ Janus 汇报：3个任务全部失败
  Completed: 0/3 tasks, 3 failed
  ── 失败详情 ──
  ❌ worker-0 — Worker crashed: ConnectionError

💡 你可以尝试：
  • 输入更具体的需求（包含文件路径、期望的输出格式）
  • 将复杂任务拆分成多个小步骤逐步执行
  • 检查任务是否依赖 web 搜索或浏览器（当前未启用）
  • 使用 'help' 命令查看使用示例和技巧
```

**新设计（~8 行）：**
```
❌ 失败 · 0/3 个任务

失败原因：
  • task-1: Worker crashed — ConnectionError
  • task-2: Worker crashed — ConnectionError  
  • task-3: Worker crashed — ConnectionError

→ 检查网络连接，输入 help 查看排错建议。
```

**删除理由：**

| 删除 | 理由 |
|------|------|
| "Janus 汇报：" | 同 2.12 |
| "Completed: 0/3 tasks" | 中文行已覆盖 |
| "── 失败详情 ──" 分隔线 | 结构上不需要额外分隔 |
| worker-N → task-N | 统一命名，用户看到的是任务 |
| 4 条建议 → 1 条指向 help | 失败原因不止一种，泛化的 4 条建议对用户帮助有限。把排错指南放到 help 中，这里只给一条明确的下一步 |

**颜色：** "❌ 失败" 红色，"0/3" 红色加粗。失败原因用红色 "✗"。

---

### 2.14 Gatekeeper 最终汇报 — 部分成功

**新设计（~6 行）：**
```
⚠️ 部分完成 · 2/3 个任务

✅ task-1: 函数正确实现
❌ task-2: Worker crashed — ConnectionError
✅ task-3: 项目目录结构完整

→ 1 个任务失败，输入 help 查看排错建议。
```

**颜色：** 标题行黄色（部分成功是警告级别），✅ 绿色，❌ 红色。

### 2.15 任务执行中 Ctrl+C 中断

**现状（2 行）：**
```
⚠ 任务被中断（按了 Ctrl+C）。
  输入新指令继续，或 'quit' 退出。
```

**新设计（1 行）：**
```
⚠ 已中断。输入新指令继续，或 quit 退出。
```

**删除理由：** "（按了 Ctrl+C）" 用户刚按的，不需要重复告知。合并到一行。

**颜色：** 黄色 ⚠ + 黄色文字。

### 2.16 运行时异常

**现状（2 行）：**
```
❌ 出错了: <exception message>
  请检查网络连接和 API 密钥，然后重试。
```

**新设计（3 行——增加一行但信息质量大幅提升）：**
```
错误: <exception message>

  可能原因：网络连接中断或 API 密钥无效。
  → 检查网络和 config.yaml 中的 api_key，然后重试。
```

**修改理由：**

| 变更 | 理由 |
|------|------|
| "❌ 出错了" → "错误:" | 减少情绪化语气，与系统错误消息风格一致 |
| 增加 "可能原因" | 帮助用户理解为什么会出错（clig.dev 错误消息 5 要素之三）|
| "请检查..." → "→ 检查..." | 箭头是明确的"下一步"信号 |

**颜色：** "错误:" 红色，"→" 黄色（行动建议）。

---

### 2.17 CLI `--help`

**现状（~12 行）：**
```
用法: python main.py [选项]

选项:
  --verbose, -v    显示详细执行日志（含模型思考过程）
  --quiet, -q      仅显示最终结果，隐藏中间步骤
  --help, -h       显示此帮助信息

示例:
  python main.py                    默认模式
  python main.py --verbose           详细模式
  python main.py --quiet             静默模式
```

**新设计（~10 行，结构一致 + ANSI 颜色）：**
```
用法: janus [选项]

选项:
  -v, --verbose    显示详细执行日志（含模型思考过程）
  -q, --quiet      仅显示最终结果，隐藏中间步骤
  -h, --help       显示此帮助信息

示例:
  janus                 默认模式
  janus --verbose       详细模式
  janus --quiet         静默模式
```

**修改理由：**

| 变更 | 理由 |
|------|------|
| `python main.py` → `janus` | 如果安装为 CLI 命令（如 `pip install -e .` 后），应使用命令名而非脚本路径 |
| 短标志移到前面 | 业界惯例：`-v, --verbose` 而非 `--verbose, -v` |

**颜色：** `janus` 加粗，选项名加粗/青色，描述白色。

### 2.18 配置文件错误（5 种场景）

**现状：** 裸文本，无颜色，无错误码。

**新设计——以 `CONFIG_NOT_FOUND` 为例：**
```
错误: CONFIG_NOT_FOUND — 配置文件未找到

  config.yaml 不在预期的位置: C:\Users\HI\Desktop\janus\config.yaml

  → 确保 config.yaml 与 janus 在同一目录下。
  → 或运行: janus init 生成默认配置。
```

**统一模板（5 种场景均用）：**
```
错误: <错误码> — <一句话标题>

  <为什么会发生>

  → <修复步骤 1>
  → <修复步骤 2（可选）>
```

**理由：** 每个配置错误获得：
1. 错误码（可搜索、可引用）
2. 原因解释
3. 具体修复步骤（不是泛化的 "请检查"）
4. 红色标题 + 黄色操作建议

## 三、信息层级重构

### 3.1 当前层级 vs 重构后

**当前（用户视角）:**
```
[启动 Banner 17 行]
[空白/提示符]
[用户输入]
[空行]
[console.summary 汇总分隔线]
[gatekeeper 最终汇报 — 与汇总信息重复计数]
```

**重构后:**
```
[启动 2 行 → 提示符]
[用户输入]
[分析阶段标题 · 青色]
[任务框 · 青色]
  [工具调用 · 暗化路径]
  [审核结果 · 绿色/红色]
[任务完成 · 耗时暗化]
[最终结果 · 绿色/红色/黄色 · 按成功/失败排序]
→ [下一步建议 · 黄色]
```

### 3.2 信息层次原则

1. **最重要信息在最后。** 用户的视线自然落在输出末尾。最终结果（通过/失败/下一步）应该是最后一个有意义的信息块。
2. **每个输出行是一条独立记录。** 方便 `grep`、`wc` 等工具处理。
3. **空行 = 一个语义分割。** 不滥用空行，当前代码中多处用了连续空行（如审核前空行），全部砍掉。
4. **表格不加边框。** 使用空格对齐——当前设计已遵守，保持不变。

## 四、进度指示重构

### 4.1 当前问题

- 没有真正的 spinner 动画——只有静态 emoji（🔍、⏳）
- "🔍 Reviewer 审核中..." 是静态文本，无法传达"正在工作"
- 静默模式只有一个 `⏳ 思考中...`，长任务时用户不知道是否卡死

### 4.2 新设计

**阶段级别进度（替代静态 emoji）：**

| 场景 | 指示方式 | 理由 |
|------|----------|------|
| 分析阶段 (< 3s) | 青色标题 `分析中…` | 太短不需要 spinner |
| 工具调用 (每次) | `⚡ 写入文件: …` | 工具调用本身就是进度 |
| 审核 (< 1s) | 无进度指示 | 瞬时完成，无需显示 |
| 长任务 (3s+) | 青色 `⏳ 执行中…` + 自动更新为耗时 | 执行完时显示实际耗时 |

**静默模式改进：**
```
⏳ 思考中…          → 初始
⏳ 思考中… (已达 10s) → 如果超过 10 秒，追加提示
完成。3/3 个任务通过。 → 最终结果
```

## 五、错误展示重构

### 5.1 当前问题

- 无错误码（无法搜索）
- 无修复建议（只有泛化的 "请检查网络连接和 API 密钥"）
- 错误消息用 `❌ 出错了:` 开头，语气偏口语化
- 未使用 `console.error()` 方法（已定义但无调用点）

### 5.2 统一错误模板

```
错误: <错误码> — <一句话描述>

  <为什么发生（1-2 句）>

  → <具体修复步骤>
  → <备选方案>

  (故障排查: https://docs.janus.dev/errors/<错误码>)
```

### 5.3 错误码定义

| 错误码 | 场景 | 当前触发点 |
|--------|------|-----------|
| `CONFIG_NOT_FOUND` | config.yaml 不存在 | main.py:143 |
| `CONFIG_PARSE` | YAML 格式错误 | main.py:145 |
| `CONFIG_MISSING_KEY` | 缺少必填字段 | main.py:157/167 |
| `ENV_VAR_NOT_SET` | 环境变量未设置 | main.py:150 |
| `API_ERROR` | API 调用失败 | main.py:322 |
| `WORKER_CRASH` | Worker 执行崩溃 | gatekeeper.py:392 |
| `DECOMPOSE_FAILED` | 无法分解任务 | gatekeeper.py:932 |

### 5.4 实现要求

- 错误信息写入 `stderr`，不污染 `stdout`
- 非 TTY 时不输出颜色但保留结构
- 堆栈跟踪仅在 `--verbose` 模式下显示
- `console.error()` 方法统一处理所有错误输出

## 八、附录：颜色工具函数签名

```python
# core/console.py 建议新增

def _supports_color() -> bool:
    """Return False when NO_COLOR is set, stdout is not a TTY, or TERM=dumb."""
    ...

def _color(code: int, text: str) -> str:
    """Wrap *text* in an ANSI SGR code. Returns plain *text* when color is suppressed."""
    ...

def _green(text: str) -> str:
    """Success / completion."""
    return _color(32, text)

def _red(text: str) -> str:
    """Error / failure."""
    return _color(31, text)

def _yellow(text: str) -> str:
    """Warning / attention."""
    return _color(33, text)

def _cyan(text: str) -> str:
    """Info / progress."""
    return _color(36, text)

def _bold(text: str) -> str:
    """Emphasis on key data."""
    return _color(1, text)

def _dim(text: str) -> str:
    """Secondary info (timestamps, metadata)."""
    return _color(2, text)
```
