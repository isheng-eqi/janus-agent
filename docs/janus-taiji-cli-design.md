# Janus 太极美学 CLI 设计方案

> 融合 Hermes CLI 研究、Janus 启动信息设计、当前美学方案，
> 以中国太极阴阳哲学重塑 CLI 视觉语言。
>
> 核心命题：**终端即宣纸，信息即墨迹。留白不是空缺，是呼吸。**

---

## 一、太极美学四原则

### 阴阳：信息与留白的平衡

> 信息是阳，留白是阴。屏幕不能塞满。

当前设计追求「极简」——删除一切可删。太极思维不同：不是越少越好，而是**该有的有，该空的白**。一段输出中，信息块是阳（实），空行是阴（虚）。阳推动理解，阴给理解留空间。

| 维度 | 极简主义 | 太极美学 |
|------|----------|----------|
| 目标 | 最少信息 | 最恰当信息 |
| 空行 | 能砍就砍 | 有意为之 |
| 密度 | 一屏尽量多 | 一屏有呼吸感 |
| 视觉重心 | 均匀分布 | 有轻有重 |

**具体规则：**
- 语义块之间必须有一行空行（阴）。两个不同信息块紧贴在一起是「阳过盛」。
- 单条信息内部不换行（阳的密度）。信息块内部的紧凑和块之间的留白形成阴阳节奏。
- 不因「可以砍」而砍空行。空行是主动设计，不是被删除后的残余。

### 动静：长任务是动，短确认是静

> 长任务有状态变化（动），短确认一笔带过（静）。动要有节律，静要有分量。

- **动（长任务执行中）：** 任务框展开、工具调用逐行出现、审核结果显示——这是阳的展开，是生长的过程。
- **静（完成确认）：** 最终结果一行落定——这是阴的收敛，是一笔写完的句号。
- **节律：** 动不能乱。工具调用的前缀、缩进、间距必须一致，形成呼吸般的节律。
- **分量：** 静不能飘。完成确认虽然只有一行，但位置（最后一行）、颜色（金色）、重量（加粗）让它稳稳落在输出底部。

### 刚柔：错误是刚，成功是柔

> 错误直接、明确（刚），成功不张扬、不抢眼（柔）。错误信息清楚但不刺眼，成功信息安静但不消失。

- **刚（错误）：** 红色但不过度饱和（避免 ANSI 亮红 `\033[91m`，用标准红 `\033[31m`）。信息结构清晰——哪错了、为什么、下一步。不吼叫，只陈述。
- **柔（成功）：** 金色而非绿色。绿色太常见太普通，金色是收敛的暖意——像印章落在纸面上，稳稳的，静静的。
- **不刚不柔（警告/进行中）：** 青色。不制造紧张，但不让你忽略。

### 黑白：终端就是黑白底

> 大面积留白，小面积点彩。颜色是点缀不是主体。

终端天然是黑底白字（或白底黑字）——这是太极图的底色。颜色只在需要引导视线时才出现，且面积要小——一个词、一个符号、一条短线。

- 正文：黑底白字（或白底黑字），不加任何颜色。
- 颜色只出现在：标题、状态标识、关键数值、下一步行动建议。
- 颜色面积占比：任何一屏输出中，彩色字符不超过总字符的 5%。

## 二、三色体系

### 2.1 从五色到三色

当前设计有 5 种语义颜色 + 2 种样式。太极方案精简为 **3 种点缀色**，黑白为底。

| 太极色 | ANSI | 命名 | 语义 | 占用 |
|--------|------|------|------|------|
| **青** | `\033[36m` | 青花 | 信息 / 进行中 / 阶段标识 | ~2% |
| **朱** | `\033[31m` | 朱砂 | 错误 / 失败 | ~1% |
| **金** | `\033[33m` | 泥金 | 成功 / 完成 | ~2% |

> 为什么是这三个颜色？
>
> **青**：取自青花瓷。中国审美中最深邃的蓝——不刺眼、不轻浮、沉静有骨。与「青出于蓝」呼应——进行中的事是成长中的事。
>
> **朱**：取自朱砂印章。中国书画中唯一的红色——不是警告的红，是「落印为信」的红。错误是盖在输出上的朱批：醒目、严肃、但不狰狞。
>
> **金**：取自泥金笺。宋画中的金不是炫耀的金，是收敛的光——像落日余晖、像旧书页边的金粉。成功不该是亮绿色感叹号，该是安静的金色印记。

### 2.2 辅助样式

不引入新颜色，只在黑白维度变化：

| 样式 | ANSI | 命名 | 用途 |
|------|------|------|------|
| **浓墨** | `\033[1m` | 加粗 | 标题、核心数值 |
| **淡墨** | `\033[2m` | 暗化 | 元数据：时间、路径、次要说明 |

> 「墨分五色」——中国传统水墨的浓淡干湿焦。终端里做不到五色，但浓淡二色已足够建立信息层次。

### 2.3 对比：当前设计 vs 太极设计

| 维度 | 当前设计 | 太极设计 |
|------|----------|----------|
| 颜色数 | 5 色 + 2 样式 | 3 色 + 2 样式 |
| 绿色 = 成功 | ✅ | ❌ 改为金色 |
| 黄色 = 警告 | ✅ | ❌ 合并到青色 |
| 红色 = 错误 | ✅ | ✅ 保留（标准红，非亮红） |
| 青色 = 信息 | ✅ | ✅ 保留 |
| 哲学来源 | 西方 UI 惯例 | 中国书画传统 |
| 颜色命名 | `_green()` / `_red()` | `_qing()` / `_zhu()` / `_jin()` |

### 2.4 颜色禁用规则（保留）

```
1. stdout 非 TTY（管道到文件或其他程序）
2. NO_COLOR 环境变量已设置
3. TERM=dumb
4. --no-color 命令行标志
```

## 三、信息密度原则：阴阳节奏

### 3.1 输出呼吸模型

```
[信息块]     ← 阳（实）
             ← 阴（虚）— 空行
[信息块]     ← 阳（实）
             ← 阴（虚）— 空行
[信息块]     ← 阳（实）
```

**每一个信息块后面跟一个空行**——不是可有可无的装饰，而是阴阳交替的呼吸。当前设计倾向于「能砍的空行都砍」（如 2.8 节删除审核前的空行），太极设计走另一条路：**空行是留白，不是浪费。**

### 3.2 信息块的重量分级

| 重量 | 视觉特征 | 场景 | 阴阳属性 |
|------|----------|------|----------|
| **重** | 浓墨 + 彩色 | 阶段标题、最终结果、错误标题 | 阳中之阳 |
| **中** | 默认样式 | 任务列表、工具调用、审核证据 | 阳 |
| **轻** | 淡墨 | 时间、路径、辅助提示 | 阳中之阴 |
| **虚** | 空行 | 信息块之间的呼吸 | 阴 |

### 3.3 屏幕占比原则

一个完整的任务执行输出（分析 → 执行 → 审核 → 结果），屏幕占比的理想状态：

```
████████░░░░░░░░░░░░░░░░  信息内容 ~35%
░░░░░░░░░░░░░░░░░░░░░░░░  留白 ~65%
```

这听起来奢侈，但太极思维下：**留白不是没信息，留白是信息的环境。** 就像一幅字画——画本身只占纸面的一小部分，但整张纸都是作品。

## 四、启动画面：阴阳平衡感

### 4.1 当前启动设计

```
**Janus · AI Agent 框架**          ← 加粗
deepseek-v4-pro · 9 工具           ← 暗化灰色
输入 help 了解用法，quit 退出。
```

**评价：** 3 行，信息完整，极简有效。但缺少「气」——太紧，像把所有东西挤在纸的最上方。

### 4.2 太极启动画面

```
                                    ← 一阴（顶部留白）
Janus                              ← 一阳（名号，浓墨）
                                    ← 一阴
deepseek-v4-pro  ·  9 工具          ← 一阳（配置，淡墨）
                                    ← 二阴（让视线落在提示符上方）
❯                                   ← 提示符（金，最后一阳）
```

**设计逻辑：**

1. **单名号，无标语。** 「Janus」三字居中——像书画作品的题签。不加「AI Agent 框架」——那是解释，不是命名。一个好的名字不需要解释。首次用户可以通过 `help` 了解细节。

2. **配置行的位置。** 不与名号紧贴——名号上方有留白，名号下方有留白——让名号悬浮于空，像印章盖在纸上。

3. **提示符是「金」。** 终端最底部，最后一抹颜色。用户视线从名号（上）→ 配置（中）→ 提示符（下），构成从上到下的阅读流。

4. **顶部留白。** 这条最简单也最重要——不把信息顶在屏幕最上方。留一行空行，让启动信息「沉」下来。

### 4.3 与 Hermes 的对比

| 维度 | Hermes CLI | Janus 太极 |
|------|-----------|------------|
| 启动信息 | Banner（边框 + logo + 工具列表） | 名号 + 配置 + 提示符 |
| 行数 | ~20+ | 4（含空行） |
| 视觉重心 | 顶部亮金 banner | 名号漂浮于空 |
| 设计哲学 | 功能清单（你有这些能力） | 意境（你在这儿） |

Hermes 的启动画面是「工具给你的清单」——这里有什么，你能用什么。Janus 太极的启动画面是「开始前的一口气」——静下来，然后开始。

## 五、任务执行中的节奏感

### 5.1 整体节奏：起承转合

太极美学下的任务执行不是线性进度条，而是一个「起承转合」的韵律：

```
                            ← 阴
 分析完成 · 3 个子任务        ← 起（青）—— 一气展开
                            ← 阴
┌─ 实现阶乘函数 ────────────┐  ← 承（青框）—— 展开
│  ⚡ 写入: factorial.py    │  ← 工具调用（淡墨路径）
│  ⚡ 执行: python test...  │
│                          │  ← 工具间的微留白
│  通过                     │  ← 转（金）—— 瞬间收束
│    ✓ 计算 5! = 120       │  ← 证据（静，默认色）
│    ✓ 边界条件正确          │
│  ⏱ 3.1s                  │  ← 时间（淡墨）
└──────────────────────────┘
                            ← 阴
 完成 · 3/3 个任务           ← 合（金，浓墨）—— 落定
                            ← 阴
```

**节奏解析：**

| 阶段 | 对应 | 阴阳 | 颜色 | 时长感受 |
|------|------|------|------|----------|
| 分析标题 | 起 | 阳 | 青 | 瞬间——一口气提起来 |
| 任务框展开 | 承 | 阳之展开 | 青框 | 展开——生成中（动） |
| 工具调用 | — | 阳之行 | 淡墨 | 逐行出现——节律 |
| 审核结果 | 转 | 阴之收 | 金/朱 | 瞬间——一笔收住 |
| 最终结果 | 合 | 阴之定 | 金浓墨 | 落定——稳稳的句号 |

### 5.2 工具调用的节律

工具调用是任务执行中最「动」的部分。它们像呼吸——一排排出现，每次都一样的格式：

```
│  ⚡ 写入: factorial.py
│  ⚡ 读取: test_factorial.py
│  ⚡ 执行: python -m pytest
```

**节律规则：**
- 前缀 `⚡` 固定，不动。这是呼吸的起。
- 动作动词（写入/读取/执行）固定 2-3 字，不动。
- 路径淡墨，是呼吸中最轻的部分。
- 每行一个工具，不合并。每行都是一次完整的吸-呼。

### 5.3 审核结果的「静」

审核通过时——不写「✅ 通过！！！！」，也不写「审核完成」。就两个字：「通过」。金色的。

```
│  通过
│    ✓ 函数正确地计算了 5! = 120
│    ✓ 代码包含类型提示和文档字符串
```

审核未通过时——也不写「❌ 失败！！！！」，就「未通过」。朱砂色的。理由紧接其后，但语气克制：

```
│  未通过
│    ✗ 函数未处理负数输入
│    ✗ 缺少文档字符串
```

**静的分量来自克制。** 说最少的话，让每个字都有重量。

### 5.4 与当前设计的节奏对比

| 节点 | 当前设计 | 太极设计 |
|------|----------|----------|
| 分析阶段 | 青色标题「分析完成 · 3 个子任务」 | 青「分析完成 · 3 个子任务」，上方留空行 |
| 任务框 | 青框 + 任务描述 | 同，但框上方留空行 |
| 工具调用 | 框内，路径暗化 | 同，路径淡墨 |
| 审核通过 | 绿「✅ 通过」 | 金「通过」，去 emoji |
| 审核失败 | 红「❌ 未通过 — 第N次重试」 | 朱「未通过」，简洁 |
| 任务完成 | 绿「✅ 通过」+ 淡墨耗时 | 金「完成」，耗时淡墨 |
| 最终汇总 | 绿/红/黄色标题 + 任务列表 | 金/朱标题 + 列表，上方留明显空行 |

## 六、结果汇报的留白

### 6.1 全部成功：安静的金色句号

```
                                    ← 阴（空行，与前文呼吸分离）

 完成 · 3/3 个任务                   ← 合（金，浓墨）

  ✓ 函数正确实现了阶乘计算              ← 证据列表（默认色）
  ✓ 测试已编写并通过
  ✓ 项目目录结构完整
                                    ← 阴（最后一口呼吸）

❯                                    ← 提示符（金）
```

**对比当前设计：**
- 当前：「✅ 完成 · 3/3 个任务」→ 绿色 + emoji。太极：金色，去 emoji。
- 当前：证据直接跟在标题下。太极：留 1 行空白。
- 当前：最后无空行直接跟提示符。太极：最后留空行让结果「沉一沉」。

### 6.2 全部失败：朱砂印记，不多说

```
                                    ← 阴

 未完成 · 0/3 个任务                  ← 合（朱，浓墨）
                                    ← 阴
 失败原因：
   • task-1: Worker 无响应 — ConnectionError
   • task-2: Worker 无响应 — ConnectionError
   • task-3: Worker 无响应 — ConnectionError
                                    ← 阴
 → 检查网络连接，输入 help 查看排错建议。  ← 下一步（金）
                                    ← 阴
```

**设计逻辑：**
- 标题不写「失败」写「未完成」——失败是判决，未完成是状态。刚而不厉。
- 失败原因列出但不吼叫。朱砂只在标题行出现，原因行用默认色。
- 下一步建议用金色——在朱砂的严肃之后，给一点暖的指引。
- 三条相同的错误全部列出——不合并、不省略。让用户看清全貌。

### 6.3 部分成功：青与金的过渡

```
                                    ← 阴

 部分完成 · 2/3 个任务                ← 合（青，浓墨——非全胜非全败，用青）
                                    ← 阴
 ✓ task-1: 函数正确实现               ← 成功项（金 ✓）
 ✗ task-2: Worker 无响应               ← 失败项（朱 ✗）
 ✓ task-3: 项目目录结构完整             ← 成功项（金 ✓）
                                    ← 阴
 → 1 个任务未完成。                     ← 下一步（淡墨）
                                    ← 阴
```

**重要变更：** 部分成功的标题颜色改黄色为青色。部分成功不是「警告」，是「进展」——有些完成了，有些还没有。青色传达「还在路上」，比黄色更准确。

## 七、逐点位对比：当前设计 vs 太极设计

以下是对照当前 `janus-cli-aesthetic-design.md` 的每个输出点位，给出太极版本。

### 7.1 启动欢迎信息

| 维度 | 当前设计（3 行） | 太极设计（4 空行间 3 信息行） |
|------|-----------------|------------------------------|
| 第一行 | `**Janus · AI Agent 框架**` | `(空行)` |
| 第二行 | `deepseek-v4-pro · 9 工具`（暗化） | `Janus`（浓墨） |
| 第三行 | `输入 help 了解用法，quit 退出。` | `(空行)` |
| 第四行 | — | `deepseek-v4-pro · 9 工具`（淡墨） |
| 第五行 | — | `(空行)` |
| 提示符 | `janus >` | `❯`（金） |

**Taiji 原则兑现：**
- 顶部留白：启动信息不顶在屏幕最上。
- 名号独立：Janus 三个字有呼吸空间。
- 配置淡化：不重要，确认即可。
- 金色提示符：唯一的颜色，让视线有一个落点。
- 去掉「输入 help 了解用法」——这是 `help` 命令的工作。不把帮助内容散落在启动画面。

### 7.2 提示符

| 当前设计 | 太极设计 |
|----------|----------|
| `janus >` | `❯`（金色） |

**理由：** `janus >` 提供了身份识别（有用），但太极设计下提示符是画面最后的落款——它应该小、轻、美。`❯` 一个字符足矣。用户知道自己打开了 Janus——刚看到启动画面。

### 7.3 空输入提示

| 当前设计 | 太极设计 |
|----------|----------|
| `输入 help 或 quit。（Ctrl+C 中断当前任务）` | `help 或 quit。`（淡墨） |

**理由：** 更短——用户在空白提示符按 Enter 时，只需要一个最短的提醒。括号内容 Ctrl+C 是重要信息，但不应占据空输入提示的位置。

### 7.4 `help` 命令

| 当前设计（~10 行） | 太极设计（~8 行） |
|--------------------|-------------------|
| `Janus — AI Agent 框架，将自然语言目标拆解为任务并逐一执行。` | `Janus`（浓墨，单独一行，上方留空行） |
| 空行 | `(空行)` |
| `用法示例：` | `自然语言描述目标，Janus 自行拆解执行。` |
| `> 帮我写一个排序 CSV 的 Python 脚本` | `(空行)` |
| `> 在 ./my-app 下创建 README.md` | `> 帮我写一个排序 CSV 的 Python 脚本` |
| `> 找出所有 TODO 注释` | `> 在 ./my-app 下创建 README.md` |
| 空行 | `(空行)` |
| `描述越具体，执行越精准。输入 quit 退出，--verbose 查看细节。` | `输入 quit 退出，--verbose 查看细节。`（淡墨） |

**Taiji 调整：** help 不是功能列表，是重逢——像翻开一本书的扉页。名号在上，简介在中，示例在下。每块之间有呼吸。去「用法示例：」标签——示例本身就是用法，不需要标签。

### 7.5 Gatekeeper 分解阶段

| 当前设计 | 太极设计 |
|----------|----------|
| `分析完成 · 3 个子任务`（青色） | `(空行)` |

**差异：** 标题上方加空行（阴），与前面的用户输入做呼吸隔离。标题和任务列表之间也加空行（阴），让标题成为一个独立的信息块。

### 7.6 任务框

| 当前设计 | 太极设计 |
|----------|----------|
| `┌─ 实现阶乘函数 ────────────┐`（青） | `(空行)` |

**差异：** 框上方加空行——每个任务框都是独立的呼吸单元。

### 7.7 工具调用

| 当前设计 | 太极设计 |
|----------|----------|
| `│  ⚡ 写入文件: factorial.py` | `│  ⚡ 写入: factorial.py` |
| `│  ⚡ 执行: python factorial.py` | `│  ⚡ 执行: python factorial.py` |

**差异：** 极小。「写入文件」缩为「写入」——与「执行」「读取」对齐词长。路径淡墨化，让动作动词和路径形成浓淡对比。

### 7.8 审核通过

| 当前设计 | 太极设计 |
|----------|----------|
| `│  ✅ 通过`（绿色） | `│  通过`（金） |
| `│     ✓ 函数正确地计算了 5! = 120` | `│    ✓ 函数正确地计算了 5! = 120` |
| `│     ✓ 代码包含类型提示和文档字符串` | `│    ✓ 代码包含类型提示和文档字符串` |
| `│     ✓ 边界条件 n=0 返回 1` | `│    ✓ 边界条件 n=0 返回 1` |

**核心变化：** 去 `✅` emoji，金色替代绿色。 `✅` 是西方式的「对了！」——太极是「通过」两个字，安静地落在框内。

### 7.9 审核失败 + 重试

| 当前设计 | 太极设计 |
|----------|----------|
| `│  ❌ 未通过 — 第 1 次重试`（红） | `│  未通过`（朱） |
| `│     ✗ 函数未处理负数输入` | `│    ✗ 函数未处理负数输入` |
| `│     ✗ 缺少文档字符串` | `│    ✗ 缺少文档字符串` |
| | `│  → 第 1 次重试`（金，另起一行） |

**核心变化：** 去 `❌` emoji。重试信息独立成行——它是行动建议（金），不与错误状态（朱）混在同一行。

### 7.10 任务完成

| 当前设计 | 太极设计 |
|----------|----------|
| `│  ⏱ 3.1s` | `│  ⏱ 3.1s`（淡墨） |

| `  ✅ 通过`（绿） | `(空行)` |
| | `  完成`（金） |

**核心变化：** 「✅ 通过」→「完成」。金色。不只是「这个任务通过了检查」，而是「这个任务圆满完成了」。

| 当前设计 | 太极设计 |
|----------|----------|
| `⚠ 已中断。输入新指令继续，或 quit 退出。`（黄） | `(空行)` |

### 7.16 CLI `--help`

## 八、进度指示：动静相生

### 8.1 阶段进度

| 场景 | 指示方式 | 颜色 | 太极逻辑 |
|------|----------|------|----------|
| 分析阶段 | `分析中…` | 青 | 起——动的开始 |
| 工具调用 | 逐行出现 `⚡` | 默认+淡墨 | 承——动的展开 |
| 审核 | 无（瞬时） | — | 转——静的瞬间 |
| 长任务（3s+） | `⏳ 执行中…` → 完成后替换为耗时 | 青→淡墨 | 动→静，过程→结果 |

### 8.2 静默模式

```
⏳ 思考中…          → 初始（青）
⏳ 思考中… (10s)    → 10 秒后追加（青，淡墨时间）
完成 · 3/3 个任务。 → 最终结果（金，浓墨）
```

静默模式不是「什么都不显示」，而是「只显示节奏中的重拍」——初始、超时提醒、最终结果。三步构成最小的起承转合。

## 九、信息层级：太极图的视觉流

### 9.1 从线性到循环

当前设计的信息层级是线性的：启动 → 输入 → 分析 → 执行 → 审核 → 汇总 → 提示符。像一条直线。

太极设计的信息层级是循环的：任何一个阶段完成后，通过空行（阴）回归平静，然后提示符（金）成为下一个循环的起点。

```
        启动（阴→阳）
           ↓
        提示符（金）
           ↓
     用户输入（阳）
           ↓
      空行（阴）
           ↓
     分析标题（青）  ← 起
           ↓
      任务框（青）   ← 承
           ↓
      审核（金/朱）  ← 转
           ↓
     最终结果（金/朱）← 合
           ↓
      空行（阴）
           ↓
    提示符（金）      ← 回到起点，等待下一个循环
```

### 9.2 视觉重心映射

```
屏幕顶部 ─── 轻（启动信息，留白多）
     ↓
屏幕中部 ─── 重（任务执行，信息密度最高）
     ↓
屏幕底部 ─── 定（最终结果，金色落款）
     ↓
提示符   ─── 轻（金点，回到起点）
```

这对应太极图的视觉结构：上方轻灵（天），中间饱满（人），下方沉稳（地）。

```python
# core/console.py

def _supports_color() -> bool:
    """Return False when NO_COLOR is set, stdout is not a TTY, or TERM=dumb."""
    ...

def _color(code: int, text: str) -> str:
    """Wrap *text* in an ANSI SGR code. Returns plain *text* when color is suppressed."""
    ...

# 三色体系
def _qing(text: str) -> str:
    """青花 — 信息 / 进行中 / 阶段标识。"""
    return _color(36, text)

def _zhu(text: str) -> str:
    """朱砂 — 错误 / 失败 / 未完成。"""
    return _color(31, text)

def _jin(text: str) -> str:
    """泥金 — 成功 / 完成 / 行动指引。"""
    return _color(33, text)

# 墨分浓淡
def _nongmo(text: str) -> str:
    """浓墨 — 标题、核心数值。"""
    return _color(1, text)

def _danmo(text: str) -> str:
    """淡墨 — 元数据、次要说明。"""
    return _color(2, text)
```

## 十二、设计决策总结

| 决策 | 结论 | 太极逻辑 |
|------|------|----------|
| 颜色数量 | 3 种点缀色（青/朱/金） | 颜色是印不是画——点缀，不是覆盖 |
| 成功色 | 金（替代绿） | 成功该是安静的暖，不是亮绿的感叹号 |
| 警告色 | 并入青 | 警告和进展是同一件事——事情在进行中 |
| emoji | 全部移除 | emoji 是西方式的情绪外化，太极是含蓄 |
| 空行 | 主动加入（阴阳呼吸） | 空行是留白，不是「砍剩的」 |
| 启动行数 | 4（含 2 空行）而非 2 | 不是最少，是最恰当 |
| 提示符 | `❯` 替代 `janus >` | 小、轻、美，像书画落款的一笔 |
| 错误标题措辞 | 保持「错误:」 | 清晰直接是刚，但不出恶声 |
| 最终结果措辞 | 「完成」「未完成」替代「通过」「失败」 | 状态而非判决 |
| 部分成功颜色 | 青替代黄 | 在路上的事，不是警告 |
| `--help` | 不修改 | 库默认行为不扭曲 |
