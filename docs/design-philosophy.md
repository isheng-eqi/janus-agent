# Janus 设计哲学：以人类管理智慧为第一设计原则

> **Janus 的第一设计原则：Agent 管 Agent 的方式，应当镜像人类管人类的方式。**
>
> 这不是隐喻，不是灵感来源——这是设计方法论。每次 Janus 面临设计决策（怎么审查？怎么重试？怎么升级？），第一个问题永远是："人类组织怎么解决这个问题？"，而不是"最高效的技术方案是什么？"。

---

## 1. 为什么是人类管理？

我们有几千年的组织智慧。军队、政府、企业、司法、制造业、学术界——每一个领域都在"怎么让一个人（或一小群人）管理一大群人的产出"这个问题上投入了海量试错。

| 领域 | 核心问题 | 年数 |
|------|---------|:--:|
| 军事 | 命令怎么传递？情报怎么上报？ | 3000+ |
| 政府 | 权力怎么委托？问责怎么追溯？ | 2000+ |
| 企业 | 目标怎么分解？信息怎么聚合？ | 200+ |
| 司法 | 判断怎么复核？标准怎么分级？ | 2000+ |
| 制造 | 质量怎么保证？缺陷怎么处理？ | 100+ |
| 学术 | 产出怎么评审？修改怎么迭代？ | 300+ |

**LLM Agent 的架构问题——任务分解、执行委派、质量审查、失败处理——和人类组织的管理问题本质上是同构的。** 差别只在于 Agent 更快、更便宜、不会抱怨，但"委托-代理问题""信息逐级失真""审查粒度""失败升级"这些结构性问题完全一样。

所以我们问：人类怎么解决的？然后抄作业。

## 2. 已应用的人类管理规则

以下 9 条规则已经在 Janus 的设计中落地。每一条都来自一个具体的人类组织实践。

### 2.1 指挥官意图（军事）→ `TaskSpec.intent`

**来自：** 军事任务式指挥（Mission Command / Auftragstaktik）。

**原理：** 上级下达意图和约束，而非详细指令。下级知道"为什么"，才能在意外情况下自主做出正确决策。

**Janus 落地：**
```python
TaskSpec.intent: str  # "为什么要做这个任务——在整体目标中的位置"
```

Worker 不只是知道"做什么"和"验收标准"，还知道"为什么"。这让 Worker 在遇到意外（API 返回格式变了、文件不存在、工具调用失败）时能做出符合意图的决策，而非机械地报错。

### 2.2 参谋/一线分工（军事）→ Gatekeeper vs Planner vs Worker

**来自：** 军事 Staff/Line 分离——指挥官（战略）+ 参谋部（战术）+ 一线部队（执行）。

**原理：** 战略思维和战术执行是两种完全不同的认知模式。让同一个人既定方向又填表格，两者都会做不好。

**Janus 落地：**
```
Gatekeeper（军师）—— 战略：理解意图、定方向、向用户汇报。零工具。
Planner（参谋）    —— 战术：拆任务、分派人、追进度、汇总结果。零工具。
Worker（士兵）     —— 执行：写代码、跑命令、读文件。有工具。
Reviewer（督察）   —— 审计：独立检查产出是否满足验收标准。零工具。
```

四个角色的职责边界清晰，每一层只看到下一层"应该看到"的信息。

### 2.3 监察长（政府）→ Reviewer

**来自：** 政府监察长（Inspector General）制度——独立于行政体系之外的审计机关。

**原理：** 做事的不能审自己的事。审查者必须独立于执行者，否则没有审计价值。

**Janus 落地：** Reviewer 是独立的 LLM 实例，不参与任务执行，不与 Worker 共享上下文，只看 TaskSpec + TaskResult。类比学术同行评审的"独立、匿名评估"。

### 2.4 质量检查点（制造）→ 每层 Review

**来自：** 制造业三道防线——原材料入厂检验（IQC）→ 在制程检查（IPQC）→ 出厂终检（OQC）。

**原理：** 缺陷越早发现，修复成本越低。每一层都应该有自己的检查点。

**Janus 落地：** Review 不是只在 Worker 完成后的收尾动作——Planner 级的 `_dispatch_with_review` 和 Worker 自分解时的子审查（`_sub_review`）都内置了审查循环。**每一层都有 Reviewer。**

### 2.5 绩效改进计划（HR）→ 携带具体反馈的重试

**来自：** 企业 HR 的 PIP（Performance Improvement Plan）——不是简单说"不合格，重做"，而是指明具体问题 + 改进标准 + 时间期限。

**原理：** "重做"本身不是管理——告诉对方"哪里不对、怎么改、做到什么标准才算过"才是。

**Janus 落地：** `_make_retry_spec()` 不只是一个重新分派——它把 Reviewer 的逐条 issue（含 severity、description）注入到重试 TaskSpec 的 context 中：
```python
feedback_parts = [review.summary]
for issue in review.blocking_issues:
    feedback_parts.append(f"[{issue.severity.value}] {issue.description}")
```

Worker 不是被"打回重做"，而是收到了一份"绩效改进计划"。

### 2.6 司法审查标准（司法）→ `ReviewVerdict` 分级

**来自：** 司法上诉的三层审查标准——重新审理（de novo）、明显错误（clearly erroneous）、滥用裁量权（abuse of discretion）。

**原理：** 不是所有类型的错误都值得同样的"推翻门槛"。有些问题一票否决，有些问题只要没有明显荒谬就接受。

**Janus 落地：** `ReviewVerdict` 五级审查结论，替换了原始的二元 pass/fail：

| 判决 | 含义 | 行为 |
|------|------|------|
| `APPROVED` | 完全通过 | 直接放行 |
| `APPROVED_WITH_NOTES` | 通过但有瑕疵 | 记录观察项，放行 |
| `MINOR_REVISIONS` | 小修 | 重试一次，自动接受 |
| `MAJOR_REVISIONS` | 大修 | 重试最多 2 次，需要重新送 Reviewer |
| `REJECTED` | 不满足核心要求 | 重试最多 2 次，否则失败 |

这直接对应学术出版的"接受 / 小修 / 大修 / 拒稿"四级制。

### 2.7 缺陷严重度分级（制造）→ `Severity` 四级

**来自：** 制造业缺陷处理矩阵——致命缺陷（停线）、严重缺陷（隔离批次）、轻微缺陷（记录后续返工）、可接受偏差（让步接收）。

**原理：** 不是所有缺陷都值得响应。致命缺陷停一切，轻微缺陷记录即可。

**Janus 落地：** `Severity` 四级：

| 等级 | 含义 | 行为 |
|------|------|------|
| `CRITICAL` | 🔴 致命——产出完全不可用 | 立即重试 |
| `MAJOR` | 🟡 严重——核心要求未满足 | 触发重试 |
| `MINOR` | 🟢 轻微——部分偏离但可用 | 重试一次，然后接受 |
| `SUGGESTION` | 💡 建议——优化想法 | 不阻塞通过 |

### 2.8 指挥链（军事）→ Gatekeeper → Planner → Worker 层级

**来自：** 军事指挥链——将军（战略）→ 上校（战役）→ 上尉（战术）→ 中士（现场）→ 士兵（执行）。

**原理：** 每一层的关注粒度不同。将军不关心士兵的瞄准镜校准数据——他只关心"高地拿下了没有"。同样，士兵不应该看到将军的全局地图——信息过载反而降低决策质量。

**Janus 落地：**
```
Gatekeeper 看到的：   架构方向、模块影响、整体通过/失败、决策点
Planner 看到的：      任务分解、Worker 分派、重试状态、升级事项
Worker 看到的：       自己的 TaskSpec、context、反馈
用户看到的：          军师汇报——只有架构级信息
```

每一层对上层做**信息压缩 + 增值解释**，不是删减——是提炼。

### 2.9 管理幅度（管理）→ 上下文纪律

**来自：** 管理学中的"管理幅度"（Span of Control）——一个管理者能有效管理的直接下属数量有上限（经典研究建议 3-7 人）。

**原理：** 超过管理幅度 → 信息过载 → 决策质量下降。上级不应该收到未经过滤的原始数据。

**Janus 落地：** `CONTEXT_DISCIPLINE_PROMPT`——Gatekeeper 的系统提示明确要求："summarize to one line"。Gatekeeper 不能也不应该看到 Worker 的工具调用细节、代码片段、日志输出。它只看到 Planner 汇总的 `ExecutionReport`——已完成任务数、失败任务数、关键摘要、升级事项。

## 3. 设计新 Janus 功能的标准流程

当 Janus 面临一个新的设计问题时，按以下四步走：

### Step 1：找到人类等价物

**问自己：这个问题在人类组织里叫什么？**

| 如果要设计…… | 人类等价物在哪个领域？ |
|-------------|---------------------|
| 多个 Worker 并行执行时需要共享信息 | 军事：通用作战图（COP） |
| Worker 之间产出冲突 | 政府：跨部门协商义务 |
| 连续多次重试失败怎么处理 | 企业：PIP 终止 → 换人或放弃 |
| Gatekeeper 如何判断一个目标该拆多细 | 企业：OKR 级联原则（每层拆 3-5 个 KR） |
| Reviewer 和 Worker 意见不一致怎么办 | 学术：编辑裁决分歧 |
| 任务执行中被高优先级任务打断 | 制造：急单插单 / 操作系统：中断优先级 |

### Step 2：提取核心机制，不是官僚流程

军队有作战司令部、有军衔、有军事法庭——但我们不需要这些。我们只需要"指挥官意图"这一个概念。

**提取原则：** 剥离制度外壳，保留核心机制。问自己三遍"为什么这个有效？"直到提炼出可迁移的核心理念。

示例：
```
军事指挥链 → 为什么有效？
  → 因为每层只处理自己层级的决策
  → 为什么这个有效？
    → 因为人脑有信息处理上限
    → 为什么这个有效？
      → 因为 LLM 的上下文窗口也有上限 → 这就是"管理幅度"在 Janus 中的等价物
```

### Step 3：映射到 Janus 角色和协议

把人类概念映射到 Janus 的具体角色和数据结构：

```
军事概念                    Janus 实现
─────────────────────────────────────────────────
指挥官意图（Commander's Intent）  → Directive.intent
参谋部（G3 Planning Staff）       → Planner
监察长（Inspector General）       → Reviewer
作战指令（OPORD）                 → Directive
战后报告（After-Action Report）   → ExecutionReport
```

### Step 4：最小化实现

只实现核心机制。不要因为人类有参谋部下属的 6 个科室（G1-G6），就让 Planner 有 6 个子模块。**先实现一个，看它是否真的过载，再拆分。**

> **一个好的 Janus 设计不是"完整模拟人类组织"——而是找到人类组织中最精简有效的那个机制，用最少的代码实现它。**

## 4. 反模式：什么时候不该用这个原则

这个原则是**灵感**，不是**模板**。以下场景不要机械套用：

### 4.1 不要因为人类有就加角色

人类组织有 HR 部门、法务部门、行政部门——Janus 不需要这些。只有当某个角色的加入解决了一个**已经存在的、可论证的问题**时，才值得增加一个角色。

**坏例子：** "企业有 PMO（项目管理办公室）来统一管理项目资源，所以 Janus 应该有一个 ResourceManager。"

**为什么坏：** Janus 当前没有多项目并行、没有资源争用。在问题不存在时设计解决方案 = 过度工程化。

### 4.2 不要加没有明确价值的流程

人类很多流程是为了**合规**（audit trail、责任分摊、法律风险），不是为了提高效率。Janus 只吸收那些**提升决策质量或执行可靠性**的流程。

**坏例子：** "企业有周报制度，所以 Planner 应该每周生成一份状态报告。"

**为什么坏：** Janus 的任务生命周期可能只有几分钟，不需要"周报"——但"执行报告"（ExecutionReport）是必要的，因为它帮 Gatekeeper 做下一步决策。

**区分标准：** 这个流程是让事情更快/更好/更可靠，还是只是让事情更"看得见"？

### 4.3 原则是灵感，不是模仿

这句话写在标题里，值得单独强调：

> **我们不是在建一个模拟人类组织的 Agent 系统。我们是在建一个 Agent 系统，从人类组织的成功和失败中学习。**

当人类实践和 Agent 的物理现实（LLM 的上下文窗口、API 调用成本、确定性 vs 概率性）冲突时——尊重 Agent 的物理现实。

**好例子：** 人类管理有一个"不越级汇报"原则——但我们允许 Gatekeeper 在 debug 模式下直接看 Worker 的原始输出，因为 Agent 不需要维护"面子"和"政治关系"。

## 5. 设计哲学总结

| 原则 | 一句话 |
|------|--------|
| **人类优先** | 先问人类怎么解决的，再问 CPU 怎么高效 |
| **提取机制** | 要"指挥官意图"这个概念，不要"军事法庭" |
| **映射角色** | 人类概念 → Janus 角色 → 最少代码 |
| **不为了像人而像人** | 不加没有明确价值的角色和流程 |
| **尊重 Agent 现实** | LLM 上下文窗口 > 人类管理理论 |
| **分层信息** | 每层看到的信息不同、信息向上流时需增值而非删减 |

---

*本文档是 Janus 的设计方法论声明。它不是一个约束——是一个思考框架。当新设计问题出现时，回来读一遍，走一遍四步流程。如果某个设计没有经过这个流程，那它可能是不必要的复杂。*
