## 一、军事：指挥链与决策循环

### 1.1 指挥链（Chain of Command）

**核心机制**：命令沿层级单向向下流动，情报沿层级向上汇总。每一层只与直接上下级交互，不允许越级指挥或越级汇报。

```
将军（战略）→ 上校（战役）→ 上尉（战术）→ 中士（现场）→ 士兵（执行）
    ↑                                    │
    └────────── 情报逐级上报 ─────────────┘
```

**为什么存在**：防止信息过载。一个指挥官无法同时跟踪 1000 个士兵的行动。层级压缩使得每个决策者只需处理可控的信息量（通常 3-7 个直接下属——这就是"控制跨度"原则的军事起源）。

**Agent 映射**：多 Agent 系统中的编排者（Gatekeeper/Orchestrator）不应直接管理所有 Worker。中间引入 Planner/TeamLead 层，每层管理 3-5 个子节点。编排者只看汇总态势，不接触 Worker 的工具调用细节。

```
Gatekeeper（战略分解）
  ├── Planner A（战术拆解）
  │     ├── Worker 1
  │     ├── Worker 2
  │     └── Worker 3
  └── Planner B
        ├── Worker 4
        └── Worker 5
```

### 1.2 任务式指挥（Mission Command / Auftragstaktik）

**核心机制**：上级下达**意图 + 约束条件**，而非详细执行指令。下级在约束范围内自主决策如何完成任务。

```
❌ 微操模式："在 14:30 精确派 3 个排从东侧进攻，用 B 队形"
✅ 任务指挥："在 15:00 前控制 X 高地，可用 A/B/C 三个排。不得使用重火力。"
```

**为什么存在**：战场信息不对称——前线指挥官比后方将军更了解当前态势。如果将军远程微操，前线会因信息延迟而错失战机。意图驱动让下级在意外情况下能做出正确决策而无需等待指令。

**Agent 映射**：TaskSpec 必须包含三个维度，不只是"做什么"：

| 字段 | 含义 | 军事对应 |
|------|------|---------|
| `description` | 做什么 | 任务目标 |
| `intent` | 为什么做（在大局中的位置） | 指挥官意图 |
| `constraints` | 不能做什么（边界条件） | 交战规则 |

Worker 知道"为什么"后，在遇到意外（文件不存在、API 返回异常格式）时能自主决策替代方案，而不是机械失败或盲目重试。

### 1.3 参谋/一线分离（Staff/Line Split）

**核心机制**：参谋军官（Staff）负责**计划和分析**，一线指挥官（Line）负责**执行和决策**。参谋不指挥部队，指挥官不亲自做情报分析。

```
指挥官（决策）  ←→  参谋长（情报整合）
  │                    ├── S1（人事参谋）
  ├── 营 A             ├── S2（情报参谋）
  ├── 营 B             ├── S3（作战参谋）
  └── 营 C             └── S4（后勤参谋）
```

**为什么存在**：认知分工——一个人无法同时做好"深度分析"和"快速决策"。参谋提供选项，指挥官做选择。两者职责不重叠，避免决策被分析污染。

**Agent 映射**：编排 Agent 不应同时做分析和决策。拆分角色：

| Agent 角色 | 职责 | 军事对应 |
|-----------|------|---------|
| Gatekeeper | 方向判断、分解决策 | 指挥官 |
| Reviewer | 独立审计、质量验证 | 参谋审计 |
| Worker | 实际执行 | 一线部队 |

Gatekeeper 零工具（不亲手执行），Reviewer 零决策（只审不判），Worker 零战略（只管执行）——三角相互制衡。

---

### 1.4 OODA 循环（Observe-Orient-Decide-Act）

**核心机制**：Boyd 提出的决策循环模型——信息不完备时有两种策略：(a) 增加信息处理能力，(b) **设计组织使其用更少信息运作**。

```
观测（Observe）→ 判断（Orient）→ 决策（Decide）→ 行动（Act）→ 观测...
```

**为什么存在**：在信息不完备且快速变化的环境中，比对手更快完成 OODA 循环的一方获胜。关键不是收集更多信息，而是在有限信息下做出"够好"的决策。

**Agent 映射**：多 Agent 系统的上下文窗口是稀缺资源。不是把所有信息塞给 LLM，而是设计信息过滤层：

- Gatekeeper 的上下文只保留**战略级信息**（任务状态、关键风险），不保留 Worker 的工具调用输出
- Worker → Gatekeeper 的回报**逐级浓缩**：Worker 输出完整结果 → Reviewer 提取关键判定 → Gatekeeper 只看到 "PASS: 3 项通过" 而非 5000 字符的代码变更
- 类比军事情报过滤："3 辆坦克在坐标 (x,y)" → "B 区敌方装甲活动" → "西侧出现装甲旅级单位"

---

### 1.5 情报逐级过滤（Intelligence Filtering）

**核心机制**：情报在上报过程中被逐级筛选和增值。每一级删除无关细节，增加判断（interpretation），最终到达高层时变成**态势判断**而非原始数据。

```
士兵 → 排长：观察到 3 辆敌方坦克在坐标 (x,y)
排长 → 连长：B 区发现敌方装甲单位活动，约排级规模
连长 → 营长：敌方在 B 区部署装甲力量，意图可能是切断我方补给线
营长 → 旅长：西侧出现敌方装甲旅级单位，建议调整防御重心
```

**为什么存在**：高层的时间用于决策而非数据处理。如果将军收到的仍是原始观测，他就得自己做从"3 辆坦克"到"装甲旅级威胁"的推理——这消耗高层带宽且容易遗漏战略视野。

**Agent 映射**：多 Agent 系统的信息传递不能只是"删减"——必须有**增值解释**。当前很多系统的 Worker → Gatekeeper 直接透传完整 TaskResult，缺少中间层增值。正确设计：

| 传递层 | 去掉什么 | 增加什么 |
|--------|---------|---------|
| Worker → Reviewer | 内部工具调用细节 | 无（原始传递） |
| Reviewer → Gatekeeper | Worker 原始产出全文 | PASS/FAIL 判定 + 关键证据摘要 |
| Gatekeeper → Planner | 单个任务判定 | 汇总态势 + 方向决策 |

## 二、政府：权力制衡与问责体系

### 2.1 三权分立（Separation of Powers）

**核心机制**：立法（制定规则）、行政（执行规则）、司法（裁决争议）三权分属不同机构，相互独立、相互制衡。

```
立法（制定规则）  ────→  行政（执行规则）
    │                        │
    └── 司法审查 ←───────────┘
         （裁决争议）
```

**为什么存在**：防止权力集中导致的滥用。同一个实体不能既制定规则又执行规则又裁决自己是否违规——这是利益冲突的最基本形式。

**Agent 映射**：多 Agent 系统的核心三角必须各自独立：

| 角色 | 权力 | 对应政府部门 |
|------|------|-------------|
| Gatekeeper | 制定 TaskSpec（规则） | 立法 |
| Worker | 执行任务（执行） | 行政 |
| Reviewer | 审计结果（裁决） | 司法 |

**关键约束**：同一 LLM 调用不能同时担任两个角色。Reviewer 不能是被审查 Worker 的同一模型实例（至少不同 system prompt），Gatekeeper 不能亲自审查自己分解的任务产出。

---

### 2.2 问责层级（Accountability Hierarchy）

**核心机制**：权威向下流，问责向上流。每一层对上一层负责，最顶层对最终用户（选民）负责。

```
选民 → 议会 → 内阁 → 部长 → 公务员
  ↑                        │
  └── 问责向上 ────────────┘
```

**为什么存在**：没有问责的权力是危险的。每一层的决策都有后果，需要有人承担。问责链确保决策质量有逐级把关。

**Agent 映射**：多 Agent 系统的问责链：

```
用户 → Gatekeeper → Planner → Worker → Reviewer
```

- **Gatekeeper 对用户负责**：最终产出质量由 Gatekeeper 承担——即使具体 bug 是 Worker 产生的
- **Worker 对 Gatekeeper 负责**：TaskResult 必须诚实标注失败，不能掩盖
- **缺失的回路**：Reviewer 的审计结果应反馈到 Gatekeeper 的分解质量——如果同一个分解模式反复被 Reviewer 打回，说明 Gatekeeper 的 TaskSpec 写得有问题（类似"部长不称职，内阁应追责"）

---

### 2.3 职能定义原则（Ultra Vires / Functional Definition）

**核心机制**：委托前必须明确定义职责边界和绩效标准。政府部门只能在法律授权范围内行事（ultra vires 原则——越权无效）。

**为什么存在**：防止部门之间的职责重叠和推诿。清晰边界 = 可追责。当出现问题时能定位到具体环节。

**Agent 映射**：TaskSpec 的 `acceptance_criteria` 就是职能定义。好的 TaskSpec 必须：
- 明确"成功"的可验证标准（不是"做好 X"，而是"X 满足条件 Y 和 Z"）
- 明确"边界"（不能碰什么——如"不要修改 test/ 目录"）
- 明确"与其他任务的接口"（依赖什么产出，产出被谁消费）

---

### 2.4 监察机制（Inspector General / Ombudsman）

**核心机制**：独立于行政体系的监察机构，拥有调查权但无执行权，直接向最高层报告。

**为什么存在**：正常层级汇报链存在"报喜不报忧"的动机——中层管理者可能掩盖问题以免影响晋升。独立的第三方监察绕开正常汇报链，提供未经过滤的真实信息。

**Agent 映射**：在多 Agent 系统中引入**独立审计层**——不同于 Reviewer（Reviewer 对 Gatekeeper 负责），监察者直接对接最终用户或系统日志：

```
正常链：Worker → Reviewer → Gatekeeper → 用户
监察链：Tracer Agent → 系统日志 → 用户（绕过正常链）
```

监察者追踪：Token 消耗是否符合预期？LLM 调用中是否有"敷衍"模式（如连续 3 轮输出相同内容）？Worker 是否在循环重试同一错误？

## 三、企业：层级管理与目标级联

### 3.1 控制跨度（Span of Control）

**核心机制**：一个管理者有效管理的直接下属数量是有限的——经典管理理论认为是 5-7 人，现代扁平组织可达 10-15，但超过后管理质量急剧下降。

**为什么存在**：认知限制。每个直接下属需要管理者的注意力进行：目标对齐、进度跟踪、障碍排除、绩效反馈。下属越多，每个人分到的注意力越少，直到管理者变成"瓶颈"。

**Agent 映射**：一个 Gatekeeper/Planner 不应并行管理超过 5-7 个子任务。超过此数时应引入中间层：

```
❌ Gatekeeper 直接管理 20 个 Worker → 审核质量崩溃
✅ Gatekeeper → 3 个 Planner → 每个 Planner 管理 3-5 个 Worker
```

对于简单任务（如文件搜索），可以放宽——因为审核成本低。对于复杂任务（如代码生成+测试），每增加一个并行 Worker 都会增加 Gatekeeper 的审核认知负担。

---

### 3.2 OKR 级联（Objectives and Key Results Cascade）

**核心机制**：目标从公司级→部门级→团队级→个人级逐层分解，每一层的关键结果（KR）对齐上一层的目标（O）。

```
公司 O：成为行业领导者
  KR1：市场份额增长 20%
    └── 部门 O：提升产品竞争力（对齐公司 O）
        KR1：推出 3 个新功能（通过对齐公司 KR1——产品增长拿份额）
          └── 团队 O：完成搜索功能重构
              KR1：搜索延迟 < 100ms
                └── 个人 O：实现倒排索引优化
                    KR1：索引构建时间减少 50%
```

**为什么存在**：确保全公司朝同一方向努力。底层工程师的日常工作与 CEO 的年度目标之间有清晰的"为什么"链路——每个层级的 KR 回答了"我的工作如何支撑上层目标"。

**Agent 映射**：多 Agent 系统的任务分解必须是**可追溯的对齐链**：

```
用户目标："给项目加完整的单元测试覆盖率"

Gatekeeper 分解：
  ├── Task A：分析现有代码结构，列出所有模块
  │     intent："为后续任务提供完整的模块清单"
  │     → 对齐用户目标：确保没有模块被遗漏
  ├── Task B：为每个模块生成测试文件
  │     intent："基于 Task A 的模块清单创建测试框架"
  │     → 对齐用户目标：覆盖率的关键是每个模块都有测试
  └── Task C：运行测试并报告覆盖率
        intent："验证 A 和 B 的产出是否真正提升了覆盖率"
        → 对齐用户目标：只有运行过才知道是否达标
```

**关键**：每个子任务的 `intent` 必须显式说明它如何支撑父目标。这不是给 LLM 的提示词——是给设计师的结构约束。

---

### 3.3 战略部署与 Catchball（Hoshin Kanri）

**核心机制**：战略目标不是单向级联，而是**上下反复协商**直到对齐。上级提方向 → 下级反馈可行性 → 上级调整 → 下级确认。

```
高层：提出方向性目标
  ↓
中层：反馈可行性评估和资源需求（"这需要 3 个团队 2 个月"）
  ↓
高层：调整目标（"那先做核心功能，次要的下季度"）
  ↓
中层：确认新目标
```

**为什么存在**：纯单向分解容易出现"不可执行的目标"——上级不了解技术约束，下级不了解战略意图。Catchball 让可行性信息向上流动，避免"接了无法完成的任务"。

**Agent 映射**：复杂/昂贵任务在分派给 Worker 之前，先做轻量级可行性检查：

```
Gatekeeper 生成 TaskSpec（草案）
  ↓
Planner/TeamLead 快速评估：Token 预估、依赖是否就绪、验收标准是否可验证
  ↓ （如不可行）
Gatekeeper 调整 TaskSpec（拆分、放宽标准、增加前置任务）
  ↓
Planner 确认 → 分派 Worker
```

这不是"让 Worker 讨价还价"，而是"在投入大量 Token 之前验证方向"。类似学术出版的 desk reject——编辑在送审前快速判断。

---

### 3.4 绩效改进计划（Performance Improvement Plan, PIP）

**核心机制**：员工绩效不达标时，不是立即解雇，而是给予明确的改进期限、具体目标和资源支持。期限到后重新评估，达标则恢复，不达标则升级处理。

**为什么存在**：人员替换成本高。培训和修正比招聘新人更经济，同时保留了组织知识。PIP 的本质是把"失败"转化为"有期限的改进机会"。

**Agent 映射**：Worker 失败不应立即丢弃任务。设计**结构化重试协议**：

```
第 1 轮失败：
  Reviewer 给出具体问题清单（不是"做得不好"，而是"缺 X、Y 不符合 Z"）
  Worker 附带修复证明重试（"✓ Fixed X: added path field to each entry"）

第 2 轮失败：
  Gatekeeper 介入判断：是 Worker 能力不够还是 TaskSpec 本身有问题？
  - TaskSpec 有问题 → 重新分解
  - Worker 有问题 → 换 Worker（不同 model/system prompt）

第 3 轮失败：
  升级给用户 → 附带完整的失败链条和已尝试的修复方案
```

---

### 3.5 授权矩阵（Delegation Authority Matrix）

**核心机制**：明确每个层级的决策权限边界。多少钱以内可以自己批？什么类型的决策必须上报？什么类型完全自主？

| 决策类型 | 个人贡献者 | 团队 Lead | 总监 | VP |
|---------|-----------|-----------|------|-----|
| 技术方案选择 | ✅ 自主 | ✅ | 知晓 | — |
| 跨团队协作 | 上报 | ✅ 自主 | 知晓 | — |
| 架构变更 | 上报 | 上报 | ✅ | 知晓 |
| 预算使用 | 上报 | 上报 | 上报 | ✅ |

**为什么存在**：防止两个极端——要么事事上报（瓶颈），要么人人自我授权（混乱）。授权矩阵在"效率"和"控制"之间取平衡。

**Agent 映射**：多 Agent 系统应定义 Worker 的决策权限边界：

| 决策类型 | Worker | Reviewer | Gatekeeper | 用户 |
|---------|--------|----------|------------|------|
| 工具选择（用什么库） | ✅ 自主 | 事后审 | — | — |
| 验收标准判定 | — | ✅ | 读结果 | — |
| 重试还是放弃 | — | 建议 | ✅ 决策 | — |
| 架构方向变更 | — | — | 建议 | ✅ 决策 |
| 目标本身是否合理 | — | — | — | ✅ |

## 四、司法：审查标准与上诉机制

### 4.1 三层审查标准（Standards of Review）

**核心机制**：上诉法院审查下级法院判决时，使用三种不同的严格度标准，取决于审查内容是什么：

| 标准 | 英文 | 适用对象 | 含义 | 推翻门槛 |
|------|------|---------|------|---------|
| **重新审理** | *de novo* | 法律问题 | 上诉法院完全不受下级结论约束 | 低——可完全推翻 |
| **明显错误** | *clearly erroneous* | 事实认定 | 只有"明确且确信地错误"才推翻 | 高——需确凿证据 |
| **滥用裁量权** | *abuse of discretion* | 程序裁量 | 只有"超出合理范围"才推翻 | 极高——极难推翻 |

**为什么存在**：司法效率——如果每个判决都被全量复审，法院系统会崩溃。分级审查让资源集中在真正重要的问题上：法律适用错误（de novo）比证据采信争议（clearly erroneous）需要更严格的审查。

**Agent 映射**：**这是多 Agent 系统最重要的审查设计原则**。不是所有验收标准都应该一视同仁：

| 验收标准类型 | 审查严格度 | 推翻门槛 | 标记 |
|-------------|-----------|---------|------|
| 安全/正确性约束 | de novo | 零容忍 | `[HARD]` |
| 功能完整性 | de novo | 零容忍 | `[HARD]` |
| 代码风格/命名 | clearly erroneous | 明显错误才打回 | `[SOFT]` |
| 架构/工具选择 | abuse of discretion | 只在明显不合理时打回 | `[SOFT]` |

**实现示例**：

```python
# TaskSpec 中的分层验收标准
acceptance_criteria = """
[HARD] 函数必须能运行且不抛异常
[HARD] 输入验证：负数输入应抛 ValueError
[HARD] 输出结果必须与参考实现的误差 < 0.001
[SOFT] 变量命名使用 snake_case 风格
[SOFT] 文档字符串使用 Google 风格
[SOFT] 函数体不超过 50 行
"""

# Reviewer 对不同级别使用不同严格度
# HARD 不满足 → FAIL，必须打回
# SOFT 不满足 → PASS_WITH_NOTES，记录观察项但不阻塞
```

**关键洞察**：如果不引入审查标准分级，Worker 会因为"变量命名不符合 snake_case"这种问题被反复打回——浪费大量 Token，且阻塞了真正重要的功能交付。

---

### 4.2 上诉流程（Appeals Process）

**核心机制**：审判结果不服 → 向上级法院上诉 → 上级法院审查（使用上述三种标准之一）→ 维持原判或推翻或发回重审。关键：**上诉法院只审下级法院审过的问题，不审新问题。**

```
初审法院（Trial Court）
  → 审理所有事实 + 适用法律
  → 判决
       ↓ 上诉
上诉法院（Appellate Court）
  → 只审法律问题（de novo）+ 事实认定是否明显错误
  → 不审新事实、不重新质证
       ↓ 再上诉
最高法院（Supreme Court）
  → 只审法律问题（de novo），不审事实
  → 选择性受理，不是每案都审
```

**为什么存在**：防止无限重审。如果每次上诉都可以引入新证据、新论点，案件永远无法终结。层级越高，审查范围越窄——从"审一切"到"只审法律问题"。

**Agent 映射**：多 Agent 系统的重试协议必须**限制重审范围**：

```
第 1 轮审查（Reviewer 审计 Worker 产出）
  → 审查范围：全部验收标准
  → 反馈：逐条问题清单（具体、可操作）

第 2 轮审查（重试后的再审查）
  → 审查范围：只审上一轮反馈中标记的问题
  → 不审 Worker 擅自修改的、与反馈无关的部分（除非引入新错误）

第 3 轮（Gatekeeper 介入）
  → 审查范围：只审"为什么连续失败"（系统性问题），不审具体代码
```

**反模式**：重试后 Reviewer 全量重新审计——如果 Worker 在第 1 轮被指出"缺 X"，第 2 轮补上了 X，Reviewer 不应该再挑 Y 和 Z 的新问题（除非 Y/Z 是新引入的错误）。这对应"上诉不审新问题"原则。

---

### 4.3 证据门槛（Burden of Proof / Evidence Standards）

**核心机制**：不同诉讼类型有不同的证据标准。刑事案件需要"排除合理怀疑"（beyond reasonable doubt），民事案件只需"优势证据"（preponderance of evidence）。

| 标准 | 所需证据强度 | 适用 |
|------|------------|------|
| 排除合理怀疑 | ~95%+ 确信 | 刑事定罪 |
| 明确且令人信服 | ~75% | 民事欺诈 |
| 优势证据 | >50% | 普通民事 |
| 合理怀疑 | ~30% | 搜查/逮捕 |

**为什么存在**：错误的代价不同。判一个无辜者有罪（false positive）比放过一个有罪者（false negative）代价大得多，所以刑事需要极高证据标准。民事纠纷只需要"一方证据更强"即可。

**Agent 映射**：Reviewer 的"判定阈值"应根据错误的代价调整：

| 场景 | 需要的证据 | 判定策略 |
|------|-----------|---------|
| 安全关键代码 | 确定性证据：运行测试 + 手动验证输出 | 没跑过测试 = FAIL |
| 功能代码 | 静态分析 + 逻辑审查 | 逻辑正确 = PASS（即使没运行） |
| 文档/注释 | 模式匹配 | 格式正确 = PASS |
| 样式/格式 | 抽样检查 | 随机抽 3 处通过 = PASS |

---

## 五、制造：质量控制与持续改进

### 5.1 三道质量防线（Three Lines of Quality Defense）

**核心机制**：质量检查分布在三个关键节点——入口、过程中、出口。

```
供应商 → [IQC：来料检验] → 生产线 → [IPQC：过程检验] → 成品 → [OQC：出货检验] → 客户
```

| 检查点 | 位置 | 发现缺陷时的处理 |
|--------|------|----------------|
| IQC（来料检验） | 原材料进入产线前 | 退货 / 让步接收 |
| IPQC（过程检验） | 生产过程中 | 停机调整 / 隔离批次 |
| OQC（出货检验） | 成品出货前 | 返工 / 报废 / 降级出货 |

**为什么存在**：缺陷越早发现，修复成本越低。在原材料阶段发现的缺陷只需退货——成本为 1x。到成品阶段发现同一缺陷，可能需要报废整批产品——成本为 100x。三道防线是成本最优策略。

**Agent 映射**：多 Agent 系统应设计三层检查，而非只在终点审查：

| Agent 检查点 | 对应制造 | 检查内容 | 发现缺陷时的处理 |
|-------------|---------|---------|----------------|
| TaskSpec 验证 | IQC | 验收标准是否可验证？依赖是否就绪？ | 重写 TaskSpec |
| Worker 执行中检查 | IPQC | Worker 是否走偏？Token 消耗是否异常？ | 提前终止 + 重试 |
| Reviewer 终审 | OQC | 验收标准是否达成？ | 通过/打回/让步接收 |

**关键**：当前大多数 Agent 系统只有 OQC（终审），没有 IQC 和 IPQC。这意味着 Worker 可能花了 50k Token 才发现 TaskSpec 本身有问题——这对应"原材料不合格但到了成品阶段才发现"。

---

### 5.2 缺陷等级分类（Defect Classification）

**核心机制**：不是所有缺陷都同等严重。致命缺陷停线，轻微缺陷记录后续处理。

| 缺陷等级 | 处理方式 | 示例 |
|---------|---------|------|
| **致命缺陷** | 立即停止整条产线 | 安全漏洞、数据损坏 |
| **严重缺陷** | 隔离该批次，通知上游 | 功能缺失、接口不兼容 |
| **轻微缺陷** | 记录并继续，后续返工 | 命名不规范、注释不全 |
| **可接受偏差** | 让步接收（concession） | 性能指标差 2% |

**为什么存在**：产线停机代价极高。如果每个小缺陷都停线，工厂一天运行不了几小时。分级处理在"质量"和"效率"之间取平衡。

**Agent 映射**：ReviewResult 不应该是二元的 pass/fail。引入五级判定：

```python
class ReviewStatus(Enum):
    PASS = "pass"                  # 完全通过，无需任何修改
    PASS_WITH_NOTES = "notes"      # 通过但有观察项（下一轮迭代关注）
    FAIL_MINOR = "fail_minor"      # 小修：Worker 自行修正，不再送 Reviewer
    FAIL_MAJOR = "fail_major"      # 大修：Worker 修改后重新送 Reviewer 审计
    FAIL_CRITICAL = "fail_critical" # 致命缺陷：停止整个任务链，升级 Gatekeeper
```

| 判定 | 是否阻塞流程 | 后续动作 |
|------|------------|---------|
| PASS | ❌ | 继续下一步 |
| PASS_WITH_NOTES | ❌ | 记录到最终报告的"观察项" |
| FAIL_MINOR | ❌（但不发下一任务） | Worker 自修，不重新送审 |
| FAIL_MAJOR | ✅ | 重新送 Reviewer |
| FAIL_CRITICAL | ✅ | 停止任务链 + 升级 Gatekeeper + 可能通知用户 |

---

### 5.3 根因分析（Root Cause Analysis / 5 Whys）

**核心机制**：不满足于修复表面症状，反复问"为什么"直到找到根本原因。丰田生产系统的"5 Whys"：一个问题连续追问 5 次"为什么"，通常能追溯到系统根源而非人为失误。

```
问题：Worker 连续 3 次产出不符合验收标准
  Why 1：因为 Worker 没有实现 [HARD] 标准的输入验证
  Why 2：因为 Worker 的系统提示词没有提到"必须验证输入"
  Why 3：因为 TaskSpec 的 description 只说"实现功能 X"没提输入验证
  Why 4：因为 Gatekeeper 的分解提示词没有要求提取隐式需求
  Why 5：因为用户只说"实现功能 X"，Gatekeeper 没有追问隐式需求
  → 根因：Gatekeeper 缺少"隐式需求提取"步骤
```

**为什么存在**：修复表面症状是打地鼠——修完一个又冒一个。根因修复一次解决一类问题。5 Whys 的本质是**沿因果链向上追溯，找到系统层面（而非执行层面）的改进点**。

**Agent 映射**：当 Reviewer 反复打回同一模式的问题时，不应只是"重试"——需要追溯到 TaskSpec 或 Gatekeeper 的分解逻辑：

```
Reviewer 记录：连续 3 个 Worker 在"错误处理"上被 FAIL
  ↓ 触发根因分析
分析结论：Gatekeeper 生成的 TaskSpec 中 [HARD] 标准没有包含错误处理
  ↓ 修复
Gatekeeper 的系统提示词增加："当任务是实现函数/API时，[HARD] 标准必须包含异常处理"
```

这对应制造业的**前馈控制（Feedforward Control）**——终检缺陷信息反馈到前端工艺改进。

---

### 5.4 持续改进（Kaizen / Continuous Improvement）

**核心机制**：不是一次性大改造，而是持续的小改进累积。丰田生产系统的核心：每个工人都可以在发现问题时拉停生产线（Andon Cord），改进建议从基层向上流动。

```
小改进 → 记录 → 标准化 → 全员推广 → 再发现新改进 → 循环
```

**为什么存在**：大改造风险大、周期长。持续小改进可以快速试错、立即验证。几万个小改进的累积效果远超一次大重构。

**Agent 映射**：多 Agent 系统不应只在"大版本"时改进。每次任务执行后自动反思：

- **Memory 层**：记录"这个 Worker 遇到 X 类型错误时用 Y 方案修好了"
- **Skill 层**：当同一模式出现 3 次以上，固化为 Skill（"处理 X 类错误的固定流程"）
- **反思层**：任务结束后评估——Gatekeeper 的分解质量如何？Reviewer 的判定准确率如何？

这对应 Hermes 的三层自我进化架构（Memory → Skill → Reflection）。

---

## 六、学术：同行评审与编辑把关

### 6.1 编辑-评审者-作者三角（Editor-Reviewer-Author Triangle）

**核心机制**：三个角色各司其职，互不越界。

```
编辑（Editor）
  ├── 选择评审者（送谁审）
  ├── 综合评审意见做最终决定（录用/修改/拒稿）
  ├── 维护期刊质量标准
  └── 处理利益冲突（不能送作者的朋友审）

评审者（Reviewer）
  ├── 独立、匿名评估稿件
  ├── 逐条提出修改意见
  ├── 给出建议（录用/修改/拒稿）
  └── 不与作者直接沟通

作者（Author）
  ├── 提交稿件
  ├── 根据评审意见逐条修改并回复
  └── 对编辑负责，不对评审者负责
```

**为什么存在**：三方制衡——编辑不亲自评审（避免偏见），评审者不最终决定（只建议不决策），作者不知道评审者是谁（确保评审者敢说实话）但可以对编辑申诉。

**Agent 映射**：多 Agent 系统的三角架构：

| 学术角色 | Agent 角色 | 职责 |
|---------|-----------|------|
| 编辑 | Gatekeeper | 选 Reviewer、读报告、做最终 PASS/FAIL 决定 |
| 评审者 | Reviewer | 逐条审计、给建议、不直接和 Worker 交互 |
| 作者 | Worker | 产出、根据反馈逐条修改、对 Gatekeeper 负责 |

**关键设计约束**：
- Reviewer 的建议不是最终决定——Gatekeeper 可以推翻 Reviewer 的 FAIL 判定（类似编辑说"评审者太严了，录用"）
- 同一 Worker 的重试应由同一 Reviewer 审计（类似期刊送同一评审者审修改稿）以保证一致性
- Gatekeeper 不应亲自审查 Worker 产出（编辑不亲自评审）

---

### 6.2 Desk Reject（编辑初筛）

**核心机制**：编辑在送审前做快速判断——稿件是否明显不合格？如果是，直接拒稿，不浪费评审者的时间。

```
投稿 → 编辑初筛 →
  ├── 明显不合格（主题不匹配/格式严重错误/学术不端）→ Desk Reject，不送审
  └── 可能合格 → 送 2-3 个评审者正式审查
```

**为什么存在**：评审者时间宝贵。据 Nature 统计，约 30-50% 的投稿在 desk reject 阶段被过滤。如果所有稿件都送审，评审系统会崩溃。

**Agent 映射**：在送 Reviewer 之前，Gatekeeper 应先做轻量级初筛：

```python
def _dispatch_with_review(spec, max_retries=2):
    result = worker.run(spec)

    # Desk Reject 检查：产出是否明显不合格？
    if _is_clearly_inadequate(result):
        # 不送 Reviewer，直接反馈重试
        # 节省一次 Reviewer 的 LLM 调用
        return _retry_with_hint(spec, "产出完全跑题/返回空/格式错误")

    # 正常送 Reviewer
    review = reviewer.review(spec, result)
    ...
```

初筛规则（高精度、低召回——宁可漏过也不能误杀）：
- Worker 返回空结果 → desk reject
- Worker 产出的文件大小 < 50 字节 → desk reject
- Worker 的 result 中没有提到任何验收标准 → desk reject
- 其他情况 → 送 Reviewer

---

### 6.3 多评审者并行（Multi-Reviewer Consensus）

**核心机制**：顶级期刊使用 2-3 个评审者独立评估同一稿件。如果意见分歧（一个说录用一个说拒稿），编辑介入判断或邀请第三个评审者打破僵局。

**为什么存在**：单一评审者有偏见——可能太严、太松、或对特定方法论有偏好。多个独立评审减少偏见。分歧本身是信号——说明稿件存在争议点，需要编辑更仔细地判断。

**Agent 映射**：关键任务使用多个 Reviewer 并行审计：

```
Worker 产出
  ├── Reviewer A（deepseek-chat + 严格 prompt）
  ├── Reviewer B（claude-sonnet + 标准 prompt）
  └── Reviewer C（deepseek-chat + 不同视角 prompt：只审架构）
        ↓
Gatekeeper 综合意见：
  ├── 全票 PASS → 通过
  ├── 全票 FAIL → 打回重试
  └── 意见分歧（2:1 或 1:1:1）
        └── Gatekeeper 介入判断（读分歧点 → 自己做决定）
```

成本权衡：不是所有任务都需要多 Reviewer。仅对以下情况使用：
- 高优先级任务（用户明确标记为关键）
- 前一轮单 Reviewer 导致争议的任务
- 验收标准涉及主观判断（如"代码是否可读"）

---

### 6.4 修改-重新提交循环（Revise-and-Resubmit Cycle）

**核心机制**：评审意见分为三级——小修（minor revision）、大修（major revision）、拒稿（reject）。

```
提交 → 编辑初筛 → 送审 → 评审意见 →
  ├── 直接录用（极罕见，<1%）→ 出版
  ├── 小修 → 作者修改 → 编辑确认 → 录用（不重新送审）
  ├── 大修 → 作者修改 → 重新送审（通常回原评审者）
  └── 拒稿 → 作者可申诉或改投
```

**为什么存在**：不是所有修改都需要重新送审。如果评审者只指出"图表 3 缺少图例"和"第 5 页有个 typo"，编辑可以自己确认修改完成——不需要再占用评审者时间。

**Agent 映射**：重试协议应按问题严重度分级：

| 问题等级 | 修改后是否需要重新送 Reviewer | 谁确认修改 |
|---------|---------------------------|-----------|
| 所有问题都是 [SOFT] 级别 | ❌ 不需要 | Worker 自检 + Gatekeeper 轻量确认 |
| [HARD] 问题 ≤ 2 个，且修改明确 | ❌ 不需要 | Reviewer 给出的修复方案明确 → Worker 照做即可 |
| [HARD] 问题 ≥ 3 个 | ✅ 需要 | 重新送 Reviewer 审计 |
| [HARD] 问题涉及架构/设计 | ✅ 需要 | 重新送 Reviewer + 可能需要 Gatekeeper 介入 |

---

## 七、跨领域通用原则

### 7.1 信息向上流：逐级浓缩，不是删减

六个领域的共性模式：每一层向上传递信息时，必须**增加解释（judgment）**，不只是删减细节。

```
                       原始信息量          传递信息量
士兵 → 排长          ████████████        ████████       (+ 判断："约排级规模")
排长 → 连长          ████████            ██████         (+ 判断："意图是切断补给线")
连长 → 营长          ██████              █████          (+ 判断："建议调整防御重心")
```

每个领域都在做这件事——只是名字不同：
- 军事：情报逐级过滤
- 企业：信息聚合金字塔
- 政府：内阁摘要
- 司法：上诉只审法律问题不审事实
- 制造：QA 报告只列缺陷不列全部检测数据
- 学术：编辑读评审意见不读原始稿件

**Agent 设计原则**：每层的信息传递接口必须定义"本级增加什么判断"——不只是"把结果传给上层"。

---

### 7.2 权威向下流：意图驱动，不是指令驱动

六个领域都发现：上层不应该对下层下达详细指令。应该下达"意图 + 约束"。

| 领域 | 向下传递什么 | 不要传递什么 |
|------|------------|-------------|
| 军事 | 指挥官意图 + 交战规则 | 具体队形、精确时间 |
| 企业 | OKR 目标 + 预算约束 | 具体行动方案 |
| 政府 | 政策方向 + 法律边界 | 执行细则 |
| 司法 | 法律标准（de novo/clearly erroneous） | 具体判决 |
| 制造 | 质量标准 + 容忍度 | 每个工艺参数 |
| 学术 | 期刊标准 + 格式要求 | 研究方法 |

**Agent 设计原则**：TaskSpec 必须包含 `intent`（为什么）和 `constraints`（边界），不只是 `description`（做什么）。Worker 需要"为什么"才能在意外情况下自主决策。

---

### 7.3 审查粒度分层：按代价决定严格度

六个领域共同发现：**错误的处理成本应该匹配错误的严重程度**。

| 领域 | 致命问题 | 严重问题 | 轻微问题 |
|------|---------|---------|---------|
| 军事 | 停战/撤退 | 调整战术 | 战后复盘 |
| 政府 | 弹劾/解散 | 调查 | 警告 |
| 企业 | 解雇/关停项目 | PIP | 谈话 |
| 司法 | 死刑/无期 | 有期 | 罚款 |
| 制造 | 停线 | 隔离批次 | 记录 |
| 学术 | 拒稿/撤稿 | 大修 | 小修 |

**Agent 设计原则**：验收标准必须标注 `[HARD]`/`[SOFT]`，Reviewer 对不同级别使用不同推翻门槛。不是所有失败都应该阻塞流程。

---

### 7.4 冗余验证：单一判断不可靠

每个领域都有冗余验证机制：
- 军事：至少两个独立情报源确认
- 政府：三权分立，法案需两院通过
- 企业：重要决策需董事会投票
- 司法：陪审团 12 人、上诉可到最高法院
- 制造：三道质量防线、双人复核
- 学术：2-3 个评审者独立评估

**Agent 设计原则**：关键任务使用多个 Reviewer（不同模型/不同 prompt）并行审计。单一 LLM 的偏见在多个独立评审中被稀释。

---

### 7.5 缺陷溯源：不要只打回，要反哺前端

六个领域都建立了"终检 → 前端改进"的反馈回路：
- 军事：战后复盘更新训练和条令
- 政府：政策失败 → 修法
- 企业：客户投诉 → 改进产品设计
- 司法：判例被推翻 → 改变法律解释
- 制造：OQC 缺陷 → 改进 IQC 和 IPQC
- 学术：评审意见 → 改进下次投稿

**Agent 设计原则**：Reviewer 的 FAIL 数据不应只用于"重试当前任务"。应该聚合分析 → 发现模式 → 改进 Gatekeeper 的分解质量或 TaskSpec 的生成模板。

---

## 附录：设计决策清单

设计多 Agent 系统时，回答以下问题：

1. **层级深度**：多少层？每层的控制跨度是多少（3-5 还是 10-15）？
2. **信息浓缩**：每层向上传递时增加什么判断？不只是数据透传
3. **意图注入**：TaskSpec 是否包含 `intent` 和 `constraints`？Worker 知道"为什么"吗？
4. **审查粒度**：验收标准是否分了 `[HARD]`/`[SOFT]`？Reviewer 对不同级别的推翻门槛是什么？
5. **缺陷分级**：失败判定是二元还是五级？什么级别的失败阻塞流程？什么级别只记录？
6. **冗余验证**：关键任务是否有多 Reviewer？用什么打破僵局？
7. **编辑初筛**：送 Reviewer 前是否有 desk reject 检查？是否在浪费评审 Token？
8. **重试协议**：重试时审查范围是否缩小（只审上次标记的问题）？还是全量重新审计？
9. **前馈回路**：Reviewer 的 FAIL 数据是否反哺到 Gatekeeper 的分解质量改进？
10. **问责方向**：每个角色的"问责对象"是否明确？最终产出由谁对用户负责？

---

> 文档基于对人类六种组织形式的系统研究：军事指挥链、政府权力制衡、企业目标管理、司法审查体系、制造质量控制、学术同行评审。六个领域在"信息浓缩""意图驱动""粒度分层""冗余验证""缺陷溯源"五项原则上高度一致——这些不是某个领域的特殊做法，而是人类大型组织管理的通用解。
